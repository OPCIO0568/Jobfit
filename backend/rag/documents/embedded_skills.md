# 임베디드 소프트웨어 직무 RAG 지식 문서

이 문서는 JobFit Agent의 로컬 RAG 검색 대상이다. 실제 기업 공고가 아니라 발표와 테스트를 위한 일반화된 임베디드 소프트웨어 직무 기준이다.

## 1. 직무별 핵심 역량

- C 또는 C++ 기반 펌웨어 구현
- MCU 주변장치 이해: GPIO, UART, SPI, I2C, ADC, PWM
- 실시간 제어와 상태 머신 설계
- 인터럽트, 타이머, 버퍼, 통신 프로토콜 처리
- 하드웨어 제약을 고려한 디버깅과 로그 분석
- 재현 가능한 테스트 절차와 핀맵 문서화

## 2. 필수 기술

- C 문법, 포인터, 구조체, 메모리 사용 이해
- MCU 개발 환경: STM32, ESP32, AVR, Arduino 계열 중 하나
- UART 로그 출력과 기본 디버깅
- GPIO 입력/출력, 타이머, PWM 제어
- datasheet 또는 reference manual 읽기
- 빌드, 플래시, 시리얼 모니터 사용

## 3. 우대 기술

- FreeRTOS task, queue, semaphore 사용 경험
- CAN, LIN, Modbus 같은 산업 또는 차량 통신 경험
- oscilloscope 또는 logic analyzer 기반 신호 확인
- watchdog, fault handling, boot mode 이해
- HIL 또는 간단한 테스트 하네스 구성 경험

## 4. 입문/중급/고급 수준 기준

- 입문: GPIO와 UART 로그로 단일 기능을 구현하고 동작을 설명할 수 있다.
- 중급: 상태 머신, 인터럽트, 통신 수신 처리, 테스트 로그를 포함해 모듈을 구현할 수 있다.
- 고급: RTOS, 통신 오류 처리, 타이밍 분석, 하드웨어 문제 분리 절차를 문서화할 수 있다.

## 5. 추천 프로젝트 예시

- 센서 로그 수집기: 센서 값을 주기적으로 읽고 UART로 CSV 또는 JSON 로그를 출력한다.
- CAN 메시지 모니터: 수신 프레임을 파싱하고 ID별 카운트, 오류 상태, 최근 값을 표시한다.
- FreeRTOS 상태 모니터: task별 상태, 주기, queue 사용량을 로그로 출력하고 과부하 상황을 재현한다.

## 6. 포트폴리오 산출물

- 핀맵과 회로 연결 설명
- 상태 머신 다이어그램
- UART 로그와 테스트 시나리오
- 오류 상황 재현 방법
- 사용한 MCU, 보드, 라이브러리 버전
- 빌드와 플래시 절차

## 7. 면접에서 검증될 수 있는 질문

- 인터럽트 안에서 오래 걸리는 작업을 피해야 하는 이유는 무엇인가?
- UART 로그가 끊길 때 어떤 원인을 의심하는가?
- CAN 통신에서 ACK 오류가 발생하면 무엇을 확인하는가?
- polling과 interrupt 방식의 차이는 무엇인가?
- 상태 머신을 사용한 이유와 상태 전이 조건은 무엇인가?

## 8. 주의사항

- 영상만 있고 코드, 로그, 핀맵이 없으면 검증 가능한 포트폴리오로 보기 어렵다.
- 하드웨어 문제와 소프트웨어 문제를 분리한 근거를 남겨야 한다.
- 보드 연결 사진에는 개인 위치, 시리얼 번호, 회사 장비 정보가 노출되지 않게 한다.
- 실제 차량, 산업 장비, 안전 관련 시스템에는 검증되지 않은 코드를 연결하지 않는다.
