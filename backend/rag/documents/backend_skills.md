# 백엔드 개발 직무 RAG 지식 문서

이 문서는 JobFit Agent의 로컬 RAG 검색 대상이다. 실제 기업 공고가 아니라 발표와 테스트를 위한 일반화된 백엔드 개발 직무 기준이다.

## 1. 직무별 핵심 역량

- 요구사항을 API, 데이터 모델, 검증 규칙으로 바꾸는 능력
- REST API 또는 GraphQL API 설계와 문서화
- 데이터베이스 모델링, 인덱스, 트랜잭션 이해
- 인증, 인가, 입력 검증, 에러 처리
- 테스트 코드와 배포 가능한 서버 구성
- 로그, 모니터링, 장애 대응을 고려한 운영 관점

## 2. 필수 기술

- 하나 이상의 백엔드 프레임워크: FastAPI, Django, Spring, Express 등
- HTTP 상태 코드, request/response, pagination, filtering
- SQL 기본: schema, join, index, transaction
- 입력 검증과 예외 처리
- 단위 테스트 또는 API 테스트
- 환경변수 기반 설정과 secret 분리

## 3. 우대 기술

- Redis, queue, background job 경험
- Docker 기반 실행 환경 구성
- OpenAPI 또는 Swagger 문서화
- CI/CD와 배포 자동화
- 부하 테스트, 캐싱, 쿼리 최적화 경험
- observability: structured logging, metrics, tracing 기본

## 4. 입문/중급/고급 수준 기준

- 입문: CRUD API를 만들고 입력 검증과 기본 테스트를 작성할 수 있다.
- 중급: 인증/인가, 데이터 모델링, API 문서, Docker 실행을 포함해 서비스를 구성할 수 있다.
- 고급: 성능 병목, 장애 대응, 캐싱, 비동기 작업, 배포와 모니터링까지 설명할 수 있다.

## 5. 추천 프로젝트 예시

- 채용 준비 관리 API: 목표 공고, 역량 갭, 로드맵을 저장하고 API 문서와 테스트를 제공한다.
- 장애 추적 티켓 API: 로그 이벤트를 저장하고 상태 변경 이력, 담당자, 우선순위를 관리한다.
- 포트폴리오 분석 백엔드: 프로젝트 README를 입력받아 기술스택, 산출물, 보완점을 구조화한다.

## 6. 포트폴리오 산출물

- API 명세서와 ERD
- 테스트 실행 결과
- 에러 응답 규칙과 입력 검증 예시
- Docker 실행 방법
- 성능 또는 쿼리 개선 전후 비교
- 운영 로그 예시와 장애 대응 메모

## 7. 면접에서 검증될 수 있는 질문

- 이 API의 데이터 모델을 왜 이렇게 설계했는가?
- 인증과 인가는 어떻게 구분했는가?
- 트랜잭션이 필요한 지점은 어디인가?
- 에러 응답 형식을 통일한 이유는 무엇인가?
- 쿼리가 느려질 때 어떤 순서로 확인하는가?

## 8. 주의사항

- 단순 CRUD만 있으면 직무 적합성을 강하게 보여주기 어렵다.
- 테스트와 문서가 없으면 실제 구현 범위를 검증하기 어렵다.
- API Key, JWT secret, DB 비밀번호를 저장소에 올리지 않는다.
- 사용자가 직접 구현하지 않은 운영 경험을 수행한 것처럼 표현하지 않는다.
