# 포트폴리오 체크리스트 RAG 지식 문서

이 문서는 JobFit Agent의 로컬 RAG 검색 대상이다. 직무별 프로젝트 산출물이 면접에서 검증 가능하도록 정리하는 기준을 제공한다.

## 1. 직무별 핵심 역량

- IT 인프라: 운영 절차, 장애 대응, 배포와 모니터링 증거
- 백엔드 개발: API 설계, 데이터 모델링, 테스트, 에러 처리 증거
- 임베디드 소프트웨어: 하드웨어 연결, 로그, 상태 전이, 테스트 절차 증거

## 2. 필수 기술

- README 작성
- 실행 방법과 환경변수 예시 제공
- 핵심 기능 캡처 또는 로그 첨부
- 테스트 방법과 결과 정리
- 한계와 개선 계획 작성

## 3. 우대 기술

- 아키텍처 다이어그램
- 요구사항과 의사결정 기록
- 장애 또는 실패 케이스 분석
- 자동화 스크립트 또는 CI 결과
- 면접 답변용 기술 선택 근거

## 4. 입문/중급/고급 수준 기준

- 입문: README, 실행 캡처, 핵심 코드 설명이 있다.
- 중급: 테스트 결과, 구조도, 실패 케이스, 개선 기록이 있다.
- 고급: 배포/운영 지표, 장애 대응, 성능 개선, 협업 기록이 있다.

## 5. 추천 프로젝트 예시

- README 중심 프로젝트 정리: 기존 프로젝트 하나를 선택해 문제 정의, 실행 방법, 결과를 보강한다.
- 테스트 보강 프로젝트: API 테스트 또는 펌웨어 검증 로그를 추가하고 실패 케이스를 문서화한다.
- 운영 산출물 보강 프로젝트: 로그, 모니터링 화면, 장애 대응 절차를 추가한다.

## 6. 포트폴리오 산출물

- `README.md`: 문제, 기능, 실행, 테스트, 한계
- `docs/architecture.md`: 구조와 데이터 흐름
- `docs/troubleshooting.md`: 오류와 해결 과정
- `docs/test-result.md`: 테스트 조건과 결과
- 실행 화면, 로그, 다이어그램 이미지
- 면접 대비 Q&A 메모

## 7. 면접에서 검증될 수 있는 질문

- README만 보고 다른 사람이 실행할 수 있는가?
- 가장 중요한 기술 의사결정은 무엇이었는가?
- 실패했던 부분과 다시 설계한다면 바꿀 점은 무엇인가?
- 테스트가 어떤 위험을 줄여주는가?
- 이 프로젝트의 한계를 솔직하게 설명할 수 있는가?

## 8. 주의사항

- 민감정보, API Key, 개인정보, 실제 회사 내부 자료를 포함하지 않는다.
- 결과만 보여주고 과정과 근거가 없으면 검증력이 약하다.
- 모든 프로젝트를 과장해 고급 수준으로 포장하지 않는다.
- 팀 프로젝트는 본인이 맡은 범위와 기여도를 분리해 적는다.
